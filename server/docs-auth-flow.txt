Backend auth flow (v1.0.2) — TeachAi
====================================

Arquivos principais
-------------------
- `config/db.js` — cria o `Pool` do `pg` usando `DATABASE_URL` e SSL.
- `controllers/authController.js` — contém as regras de negócio de registro e login.
- `routes/routes.js` — expõe as rotas HTTP `/api/auth/register` e `/api/auth/login`.
- `src/app.js` + `index.js` — configuram o Express, JSON, CORS e sobem o servidor.


config/db.js — conexão com Postgres
-----------------------------------
- Importa `dotenv` e `pg`.
- Lê `process.env.DATABASE_URL`.
- Cria um `Pool` com SSL habilitado (Neon/Postgres gerenciado).
- Exporta `pool`, que é usado pelos controladores para executar queries.


src/app.js — app Express básico
-------------------------------
- Cria um `app` Express, aplica `cors()` e `express.json()` para aceitar JSON.
- Usa o roteador principal em `/api`:
  ```js
  app.use("/api", routes)
  ```
- Não contém lógica de negócio, só configuração de middlewares e rotas.


index.js — inicialização do servidor
------------------------------------
- Carrega variáveis de ambiente com `dotenv.config()`.
- Importa `app` e `pool`.
- Antes de iniciar o servidor, testa a conexão com o banco:
  - `await pool.query("SELECT 1")`.
  - Se der certo, loga “✅ Connected to database” e sobe o servidor na porta configurada.
  - Se falhar, loga o erro e encerra o processo.


routes.js — definição das rotas de autenticação
-----------------------------------------------
- Importa `Router` do Express, `health` e os handlers `register` e `login`:
  ```js
  router.get("/health", health)
  router.post("/auth/register", register)
  router.post("/auth/login", login)
  ```
- Essas rotas ficam sob o prefixo `/api` por causa do `app.use("/api", routes)`:
  - Registro: `POST /api/auth/register`
  - Login: `POST /api/auth/login`


authController.js — registro
----------------------------
Fluxo do `register(req, res)`:

1. Lê `name`, `email`, `password` do corpo (`req.body`).
2. Valida campos obrigatórios:
   - Se faltar nome, e-mail ou senha, responde `400` com mensagem amigável.
3. Valida força da senha com regex:
   - Mínimo 8 caracteres.
   - Deve ter letra maiúscula, minúscula, número e símbolo.
   - Se não passar, responde `400` com mensagem explicando o requisito.
4. Verifica se já existe usuário com aquele e-mail:
   - `SELECT id FROM "User" WHERE email = $1`.
   - Se encontrar, responde `409` com mensagem de “email já existe”.
5. Gera hash seguro da senha com `bcrypt.hash(password, 10)`.
6. Insere o novo usuário na tabela `"User"`:
   - `id` é gerado no banco com `gen_random_uuid()::text`.
   - `createdAt` e `updatedAt` são preenchidos com `NOW()`.
   - A query retorna `id`, `name`, `email`, `createdAt`.
7. Responde `201` com:
   ```json
   {
     "user": { ... },
     "message": "Usuário registrado e logado com sucesso."
   }
   ```
   - O frontend interpreta isso como “login automático” após o registro.

Qualquer erro inesperado cai no `catch`, é logado no servidor e responde `500` com uma
mensagem genérica para o cliente.


authController.js — login
-------------------------
Fluxo do `login(req, res)`:

1. Lê `email` e `password` do `req.body`.
2. Valida obrigatórios:
   - Se faltar e-mail ou senha, responde `400`.
3. Busca o usuário pelo e-mail:
   - `SELECT id, name, email, password FROM "User" WHERE email = $1`.
   - Se não encontrar, responde `401` (“Email ou senha inválidos.”).
4. Verifica a senha armazenada:
   - Há suporte para dois formatos:
     - Hash `bcrypt` (strings que começam com `$2`).
     - Senhas legadas em texto puro (para usuários criados antes do bcrypt).
   - Se for hash, usa `bcrypt.compare(password, storedPassword)`.
   - Se for texto puro, compara diretamente `password === storedPassword`.
5. Se a senha não confere, responde `401` com mensagem de erro.
6. Se a senha confere:
   - Remove o campo `password` do objeto retornado.
   - Responde `200` com:
     ```json
     {
       "user": { "id": "...", "name": "...", "email": "..." },
       "message": "Login realizado com sucesso."
     }
     ```

Assim, o frontend não precisa conhecer detalhes do banco; ele apenas envia
`email`/`password` e recebe `user` em caso de sucesso.
