FormRegister.jsx + FormLogin.jsx + fluxo de navegação — lógica atual
====================================================================

VISÃO GERAL
-----------
- O frontend não usa mais `db.json` para autenticação.
- Tanto registro quanto login conversam com a API real (`/api/auth/register` e `/api/auth/login`),
  que valida e persiste os dados no Postgres.
- Após sucesso (registro ou login), o app navega para `/home` passando o usuário autenticado
  no `state` da rota (`navigate("/home", { state: { user } })`).


FormRegister.jsx — criação de conta + “login automático”
--------------------------------------------------------
- Estados principais:
  - `name`, `email`, `userPassword`: controlam os inputs do formulário.
  - `loading`: indica se há requisição em andamento (desabilita o botão e troca o texto).
  - `error`: mensagem de erro vinda da API (email já existe, senha fraca, etc.).

- Submit (`handleSubmit`):
  1. `e.preventDefault()` — impede o reload padrão do navegador.
  2. Zera `error` e liga `loading`.
  3. Chama a função de API `registerUser({ name, email, password: userPassword })`.
     - Essa função faz `POST /api/auth/register` usando Axios.
  4. A API, no sucesso, devolve:
     ```json
     {
       "user": { "id": "...", "name": "...", "email": "...", "createdAt": "..." },
       "message": "Usuário registrado e logado com sucesso."
     }
     ```
     O componente extrai `const user = data.user || data`.
  5. Limpa os campos do formulário.
  6. Faz o “login automático”: `navigate("/home", { state: { user } })`, mandando o usuário
     autenticado para a Home.
  7. Em caso de erro (`catch`), lê `err.response?.data?.message` se existir (vinda do backend)
     ou mostra uma mensagem genérica de falha. Essa mensagem vai para `error` e é renderizada
     na interface.

- Validações de negócio (lado servidor, mas impactam o front):
  - Verifica se `name`, `email` e `password` foram enviados.
  - Verifica se o e-mail já existe na tabela `"User"`.
  - Verifica se a senha atende aos critérios de força:
    - Mínimo 8 caracteres.
    - Pelo menos 1 letra minúscula, 1 maiúscula, 1 número e 1 símbolo.
  - Em qualquer falha, o backend devolve `400` ou `409` com mensagem explicando o problema,
    que o front mostra em `error`.


FormLogin.jsx — login com API e navegação
-----------------------------------------
- Estados:
  - `email`, `userPassword`: inputs controlados.
  - `error`: mensagem de erro (por exemplo, “Email ou senha não conferem.”).
  - `loading`: carrega enquanto a API de login está sendo chamada.

- Submit (`handleSubmit`):
  1. `e.preventDefault()`.
  2. Zera `error` e liga `loading`.
  3. Chama `loginUser({ email, password: userPassword })`, que faz `POST /api/auth/login`.
  4. Se a API retornar sucesso (`200`), espera um corpo do tipo:
     ```json
     {
       "user": { "id": "...", "name": "...", "email": "..." },
       "message": "Login realizado com sucesso."
     }
     ```
     - O componente extrai `user` e limpa os campos.
     - Em seguida, navega para `/home` com `navigate("/home", { state: { user } })`.
  5. Se a API devolver `401` ou outro erro, o `catch` lê a mensagem em
     `err.response?.data?.message` (por exemplo, “Email ou senha inválidos.”) ou usa
     uma mensagem genérica. Essa string vai para `error` e aparece abaixo dos inputs.
  6. `loading` volta para `false` no `finally`.


Botões e feedback visual
------------------------
- No registro:
  - Botão mostra `"Registering..."` quando `loading` é `true`, e `"Register"` caso contrário.
  - O botão fica `disabled` enquanto `loading` é verdadeiro para evitar múltiplos submits.

- No login:
  - Botão mostra `"Logging in..."` enquanto `loading` é `true`, e `"Login"` no estado normal.
  - Também é desabilitado durante a chamada.

- Mensagens de erro:
  - São renderizadas somente se `error` não estiver vazia.
  - No login, é exibida com estilo de texto em vermelho e `role="alert"` para acessibilidade.


Fluxo geral de navegação
------------------------
1. **Registro**:
   - Usuário preenche nome, e-mail e senha forte.
   - Frontend chama `/api/auth/register`.
   - Backend cria o registro no banco (com id gerado e senha hasheada com bcrypt).
   - Resposta inclui o `user`; o front faz “login automático” e redireciona para `/home`.

2. **Login**:
   - Usuário preenche e-mail e senha.
   - Frontend chama `/api/auth/login`.
   - Backend valida e-mail + senha (comparando com hash ou senha antiga, se for o caso).
   - Se ok, devolve `user` e o front navega para `/home`.
   - Se não, devolve `401` com mensagem, exibida no `FormLogin`.
